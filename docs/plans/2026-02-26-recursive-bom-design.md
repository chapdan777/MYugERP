# Рекурсивная структура изделий (Recursive BOM)

## Описание проблемы

Текущая система не позволяет связать **Дополнительные свойства (ДС)** с **Операциями** при создании номенклатуры. Кроме того, одна и та же операция (например, «Фрезеровка фаски») может применяться к разным деталям изделия (филенка, фасад) с разными размерами и формулами расчёта.

## Принятое решение

**Рекурсивная спецификация (BOM)** — каждая деталь является отдельной **Номенклатурой** со своими операциями, материалами и маршрутом. Изделия вкладываются друг в друга через окно «Деталировка».

## Реальный пример: Фасад с филенкой

```
Фасад (H=1000, W=600)                       ← Операция: Сборка
├── Профиль 1 (верхний):   H=W, W=W_PR1     ← Операции: Раскрой, Фрезеровка паза
├── Профиль 2 (левый):     H=H, W=W_PR2     ← Операции: Раскрой, Фрезеровка паза
├── Профиль 3 (нижний):    H=W, W=W_PR3     ← Операции: Раскрой, Фрезеровка паза
├── Профиль 4 (правый):    H=H, W=W_PR4     ← Операции: Раскрой, Фрезеровка паза
└── Филенка                                  ← Операции: Фрезеровка фаски, Шпонирование
    H = H - (W_PR2 + W_PR4) + GP * 2
    W = W - (W_PR1 + W_PR3) + GP * 2
    └── Рубашка (шпон 3мм)                  ← Операции: Раскрой раскрой
        H = H - 10,  W = W - 10
```

Где `W_PR1`…`W_PR4`, `GP` — **ДС фасада**, доступные в формулах.

## Ключевые принципы

### 1. Область видимости (Scoped Context)

Каждый уровень видит только **свои** `H`, `W`, `D`. Родитель вычисляет размеры дочернего элемента по формулам, и результат **становится** `H`/`W`/`D` дочернего контекста.

### 2. Дополнительные переменные

Родитель может передать кастомные переменные (например, `GP`, `Zazor`) через `extra_variables`. Они будут доступны в контексте дочернего элемента.

### 3. Условное включение

Компонент может включаться условно: `condition_formula = "ПРИКЛЕЙКА_ДЕКОРА == 'true'"` — рубашка добавляется только если выбран ДС «Приклейка декора».

## Архитектура изменений (Фаза 1: CRUD + UI)

### Таблица `product_component_schemas` (расширение)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | PK | — |
| `product_id` | FK → products | Родительское изделие |
| **`child_product_id`** | FK → products, nullable | Дочерняя номенклатура (null = простая деталь) |
| `name` | varchar | Название компонента |
| `length_formula` | text | Формула H дочернего элемента |
| `width_formula` | text | Формула W дочернего элемента |
| **`depth_formula`** | text, nullable | Формула D дочернего элемента |
| `quantity_formula` | text | Формула количества |
| **`extra_variables`** | jsonb, nullable | `{"GP": "10", "Zazor": "5"}` |
| **`condition_formula`** | text, nullable | Условие включения |
| **`sort_order`** | int, default 0 | Порядок отображения |

### Бэкенд

- **`ProductComponentSchema`** (Domain) — добавить поля.
- **`ProductComponentSchemaEntity`** (Persistence) — добавить колонки.
- **`CreateProductComponentSchemaUseCase`** — валидация `childProductId`, защита от циклов.

### Фронтенд

- **`ProductForm.tsx`** — добавить кнопку «Деталировка».
- **`ProductComponentsDialog.tsx`** — Autocomplete для выбора номенклатуры, поля формул, секция «Доп. переменные», поле условия.

## Фаза 2 (будущее): Рекурсивная генерация ЗН

При генерации ЗН система рекурсивно обходит дерево компонентов, вычисляет размеры на каждом уровне и собирает плоский список операций.

## План верификации (Фаза 1)

1. Открыть номенклатуру «Фасад» → Деталировка.
2. Добавить компонент → выбрать номенклатуру «Филенка».
3. Указать формулы `H - (W_PR2 + W_PR4) + GP * 2`.
4. Сохранить и проверить данные в БД.
